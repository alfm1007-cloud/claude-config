# 5단계 검증 — Phase G v63 박제 정책

지금 변경한/변경할 코드에 대해 **반드시 5단계 끝까지 추적**한 뒤 결과를 commit/push 메시지에 포함하세요. v62 +2P 사고 같은 중복 호출 버그는 3단계 한 번만 했어도 막혔을 일.

## 출력 형식 (그대로 채워서 답변)

```
5단계 검증:
1) 변경 지점: <파일:함수:라인 — 무엇을 바꿨는지>
2) 데이터 흐름: <입력값 → 자동변환 → 저장 → 표시까지 끝까지 1회 완주>
3) 중복 호출 검사: <같은 효과(addPoints/upsert/delete 등) 다른 경로 grep 결과>
4) edge 케이스: <0 / null / 빈문자열 / 음수 / 동시호출 — 각각 처리 명시>
5) 사용자 행동 재현: <화면에서 어떻게 보일지 클릭 시뮬레이션 1회>
```

## 자동 grep 명령 힌트
- 중복 호출: `Grep("addPoints\\(|sb\\.from\\('point_history'\\)\\.insert", glob: "public/js/*.js", output_mode: "content")`
- update 누락: `Grep("\\.update\\(\\{[^}]*<필드명>", glob: "public/js/*.js")`

## 사용 사례
- 사용자 사전: "방금 sales.js 수정했는데 5단계 검증해줘"
- 사용자 사후: "/5step" → 마지막 변경에 대해 검증 보고

## 절대 잊지 말 것
**"3단계(중복 호출 검사) 한 번만 했으면 잡혔을 버그"** — v62 사례.
