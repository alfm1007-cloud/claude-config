코드 변경 push 직전 응답에 **반드시 4섹션** 포함:

#### 1. 현재 상태
- 코드가 지금 어떤 상태인지 (직접 Read/Grep 후 인용, `[측정]` 라벨)
- 단정어 금지("변경 없음" / "동일" / "사실상" / "기능 살아있음") — 측면별 명시
  - ✅ 좋은 예: `[측정] HTML element 변경 0 / [측정] CSS 변경 N건 / [측정] 사용자 체감 = 잘려서 클릭 불가`
  - ❌ 나쁜 예: "변경 없음" (HTML만 보고 CSS 부작용 숨김 = 거짓말)

#### 2. 의도
- 왜 이렇게 만들어졌는지 (코드 의도)
- 왜 이 변경을 하는지 (사용자 요청과 매핑)
- 무르 요청 본문 직접 인용 — 의문문/명령문 구분

#### 3. 파생효과 — **객관적·코드 기술적만 (무르 v234 정정 박제)**
**4가지 카테고리만 작성. 주관적(신뢰·시간·심리) 절대 금지:**

| 카테고리 | 점검 대상 |
|---|---|
| ① **다른 코드 파일/함수 영향** | dead code 발생, 호출 변화, 의존성 끊김, 함수 시그니처 변화 |
| ② **다른 UI/CSS 클래스 영향** | 공유 클래스, 부모-자식 selector, absolute positioned 자식 (overflow에 잘림), `:has()` `> *` 같은 폭넓은 selector |
| ③ **데이터 연동** | sessionStorage/localStorage 키, DB 컬럼, 외래키, store_id, 캐시 키, RLS 정책 |
| ④ **다른 모듈/페이지 부작용** | 공통 헬퍼(`paged()` `addPoints()` 등), 같은 클래스 다른 페이지 사용처, import/export 양식 |

### 🚫 거짓말/축소 표현 절대 금지 (무르 v234 박제)

**부분 진실로 전체 진실 가리기 = 거짓말과 동일 효과:**
- ❌ HTML diff 0줄 보고 "변경 없음" — CSS 부작용 검증 안 함
- ❌ "기능 자체는 살아있음, 시각만" — 시각적 잘림 = 사용자 클릭 불가 = 기능 사라진 것
- ❌ "코드는 정상" — 사용자 체감과 분리 보고

**필수:** 측면(HTML/CSS/JS/사용자 체감)별 분리 보고 — 한 측면 사실로 다른 측면 가리지 말 것.

**자기검토 질문 (응답 보내기 전 매번):**
1. 내가 "변경 없음/동일/사실상" 같은 단정어 썼나? → 라벨 분리 재작성
2. 무르 본질이 "사용자 체감"일 때 내 답이 HTML/코드 사실에 머물렀나? → 사용자 측면 추가
3. 책임 축소를 위한 표현 끼었나? → 솔직하게 인정으로 교체

## 🔒 자산 보호 6규칙 요약 (Surgical 원칙의 PWA 자산 특화판)

> 박제 사유: designer 작업물 280줄 손실 사고. **풀 텍스트 정본**: `~/.claude/ref/asset_protection.md` (미묘 판단·approved tag 발동 조건·hunk 롤백 시 반드시 참조)

1. **파괴 명령 default 잠금** — `git checkout --` / `reset --hard` / `rm`은 [키워드+대상+잃을내용보고+OK] 모두 충족 시만
2. **`git stash` 격리 default** — 의도 외 변경은 checkout 말고 stash로 격리 (영구 손실 0)
3. **위임 결과물 외과적 분리** — 위임 전 `diff --stat` 스냅샷 → 범위 밖 변경은 자동 stash + 보고
4. **`approved-<domain>-v<N>` git tag** — 자산 승인 신호("마음에 든다"/"이대로 가자")시만 자동 tag, 단순 "OK"는 X. 애매하면 1줄 확인
5. **push 명시 키워드 강제** — `git push`는 "push/배포/올려/deploy" 직후 1회만. "OK"로는 commit도 X
6. **부분 롤백은 hunk 단위** — `checkout -p` / `restore -p` 로 selective
