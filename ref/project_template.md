# 프로젝트 CLAUDE.md 표준 템플릿

> 무르가 **"이 프로젝트 CLAUDE.md 만들어줘"** 라고 말하면 Claude는 이 템플릿을 뼈대로 쓴다.
> 성내동/CLAUDE.md와 swresort/CLAUDE.md가 이 구조를 따른다.
> **채우기 규칙**: 모르는 건 `*(확인 필요)*` 라벨. 감으로 채우지 말 것.
> **분량 상한**: 각 섹션 20줄 이내 권장. 넘으면 별도 `.claude/ref/*.md`로 분리.

---

## 📋 13섹션 뼈대

```markdown
# CLAUDE.md — [프로젝트 이름]

> 전역 규칙은 `~/.claude/ref/rules.md` 참조.
> 인프라 전반은 `~/.claude/memory/infrastructure.md` 참조.
> 이 파일은 [프로젝트] 고유 약속만 박제한다.

## 1. 프로젝트 정체성
- 용도 / 운영 주체 / 규모 / 핵심 기능
- 배포 URL
- 로컬 경로
- GitHub 레포

## 2. 사용자 전제 (비즈니스 결정 — 바꾸려면 무르 승인)
| 항목 | 결정 | 이유 |
예) 외상=없음, 회원권 개념=없음, SMS=도입 안 함

## 3. 기술 스택 + 파일 구조
| 레이어 | 기술 |
+ 디렉토리 트리 (핵심만)

### 3-1. 환경변수 & 시크릿
| 키 | 용도 | 어디서 받는가 |
- .env 위치, 하드코딩 허용/금지 키

### 3-2. 로컬 실행법
1. Node/npm 버전 요구
2. `npm install`
3. `.env` 설정
4. 실행 명령 (`npm start` / `npm run dev`)
5. 실패 시 섹션 9 참조

## 4. 핵심 패턴 (기술 결정 — 리팩터 가능, 근거 있으면)
예) XSS 방어, N+1 제거, TTL 캐싱, 이벤트 위임

## 5. 0원칙 요약
> 전문은 `~/.claude/ref/rules.md`
- 핵심: "현재 상황 확인 → 근거 인용 → 제안"
- 금지: 확인 없이 추측 기반 제안

## 6. 개발 흐름

### 6-1. 배포 체크리스트
1. 코드 변경 → commit
2. 🔴 `git push` (자주 빼먹음)
3. 자동 배포 대기
4. 성공 확인 (Deploys/Logs)
5. PWA 재시작 (해당 시)

### 6-2. 롤백 전략
- 플랫폼 재배포 (Render/Netlify Deploys → 이전 성공 버전)
- DB: 역마이그레이션 SQL (스키마 변경이 원인일 때)
- 긴급: `git revert <해시>` → push

### 6-3. 커밋 컨벤션
- `feat(범위): 제목` / `fix(범위): 제목` / `refactor(범위): 제목` / `docs: 제목`

## 7. 관측 / 로그
| 종류 | 위치 |
- 서버 로그 / DB 콘솔 / 프론트 에러 / keep-alive 상태 / 알림 채널

## 8. 외부 의존성 위험도 (선택)
| 서비스 | 장애 시 영향 | 대체 경로 |
예) Supabase=치명(대체 없음), Claude API=기능 비활성 폴백

## 9. 자주 발생 오류 (프로젝트 전용)
| 오류 | 원인/해결 |
- 범용 오류는 `~/.claude/ref/errors.md` 중복 기재 금지

## 10. 보류 / 미확인
| 항목 | 상태 | 언젠가 할 일 |

## 11. 최근 변경 이력 (최대 3개)
- 그 이전은 git log / GitHub Releases로. 섹션당 15줄 상한.

## 12. 담당자
- 이름 · 이메일 · UID · GitHub

## 13. 업데이트 규칙 (메타)
- 스택 변경 → 섹션 3 갱신
- DB 컬럼 변경 → 섹션 4 또는 별도 db.md 포인터
- 신규 오류 해결 → 섹션 9 추가
- 완료 페이즈 → 섹션 11에 추가, 3개 넘으면 오래된 것 제거
```

## 🔑 섹션 2 vs 4 경계 (혼동 방지)

| | 섹션 2 "사용자 전제" | 섹션 4 "핵심 패턴" |

| 성격 | **비즈니스/운영 결정** | **기술 구현 결정** |

| 질문 방식 | "~ 방향 바꿀까요?" 먼저 | "대안 제안드려요" + 비교표 |
